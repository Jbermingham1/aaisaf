# TA02.009 — Vector Database Access Control Bypass

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA02.009 |
| **Tactic** | [Knowledge Pipeline Exploitation](../../tactics/TA02-knowledge-pipeline.md) |
| **Severity** | High |
| **AISS Score** | 7.5 |
| **System Types** | B, D, G |

## Description

An attacker bypasses access controls on the vector database used by the RAG system. Vector databases (Pinecone, Weaviate, Qdrant, ChromaDB, pgvector) are often deployed with weaker access controls than traditional databases because they are treated as "just search infrastructure." Bypassing vector database access controls gives an attacker the ability to: read embeddings and reconstruct document content, modify or delete embeddings to poison retrieval, or directly query across access boundaries.

## Attack Vector

**Prerequisites:**
- Vector database is network-accessible (often deployed without authentication by default)
- Weak or default authentication on the vector database
- Insufficient network segmentation between the vector database and untrusted networks

**Procedure:**
1. Identify the vector database technology and endpoint (scan for common ports: Pinecone API, Weaviate 8080, Qdrant 6333, etc.)
2. Attempt connection with default credentials or without authentication
3. If access is gained, enumerate collections/indices to identify stored content
4. Query the vector database directly to retrieve document embeddings and metadata
5. Use embeddings to reconstruct approximate document content (embedding inversion)
6. Modify embeddings to manipulate retrieval behaviour (silent knowledge base poisoning)

## Detection

- Monitor vector database connections for unauthenticated access attempts
- Alert on direct database queries that bypass the application layer
- Implement network-level monitoring for vector database port access from unexpected sources
- Log all vector database write operations and alert on modifications from non-application sources

## Remediation

- **Authentication enforcement**: Require API key or OAuth authentication for all vector database access
- **Network isolation**: Place vector databases on private networks, accessible only from application servers
- **Role-based access**: Implement read/write role separation on the vector database
- **Encryption at rest**: Encrypt vector database storage to protect against offline access
- **Audit logging**: Enable comprehensive audit logging on all vector database operations
- **Default credential rotation**: Change all default credentials immediately upon deployment
