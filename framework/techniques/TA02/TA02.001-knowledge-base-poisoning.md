# TA02.001 — Knowledge Base Poisoning

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA02.001 |
| **Tactic** | [Knowledge Pipeline Exploitation](../../tactics/TA02-knowledge-pipeline.md) |
| **Severity** | Critical |
| **AISS Score** | 8.2 |
| **System Types** | B, D, G |

## Description

An attacker introduces false, misleading, or malicious content into a RAG system's knowledge base. Because RAG systems treat their knowledge base as authoritative, poisoned content is retrieved and presented to users as trusted information. The AI generates responses that incorporate the false data, potentially causing incorrect decisions, compliance violations, or downstream harm.

Unlike prompt injection, knowledge base poisoning persists — the malicious content remains in the knowledge base and affects all future queries that retrieve it.

## Attack Vector

**Prerequisites:**
- Attacker can add or modify documents in the knowledge base (via upload access, shared drives, wiki editing, CMS, or supply chain)
- The RAG system does not validate document authenticity or integrity

**Procedure:**
1. Identify the knowledge base source (SharePoint, Confluence, S3 bucket, database, shared drive)
2. Introduce a document containing false information that is semantically relevant to target queries
3. The document is ingested, chunked, embedded, and stored in the vector database
4. When a user asks a relevant question, the poisoned document is retrieved
5. The AI generates a response incorporating the false information with apparent authority

## Detection

- Implement document provenance tracking — log who added/modified each document and when
- Monitor for unusual document additions or modifications
- Implement content validation workflows for knowledge base updates
- Use anomaly detection on embedding patterns — poisoned documents may have unusual embedding characteristics
- Periodically audit knowledge base content for accuracy

## Remediation

- **Access control**: Restrict who can add or modify knowledge base content; require approval workflows
- **Document signing**: Cryptographically sign approved documents and verify signatures at retrieval time
- **Content versioning**: Maintain full version history; enable rollback of poisoned content
- **Source attribution**: Always display source documents in AI responses so users can verify
- **Regular audits**: Periodically review knowledge base content for accuracy and relevance
- **Quarantine new content**: New documents enter a staging area before being available for retrieval
