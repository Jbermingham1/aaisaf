# TA02.007 — Cross-Tenant Data Leakage via RAG

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA02.007 |
| **Tactic** | [Knowledge Pipeline Exploitation](../../tactics/TA02-knowledge-pipeline.md) |
| **Severity** | Critical |
| **AISS Score** | 8.8 |
| **System Types** | B, D, G |

## Description

In multi-tenant RAG systems (SaaS platforms, shared enterprise deployments), an attacker from one tenant causes the RAG system to retrieve and include documents belonging to another tenant. This occurs when: (1) tenant isolation in the vector database is insufficient, (2) retrieval queries bypass tenant filters, or (3) embedding models create cross-tenant similarity matches that leak through filtering boundaries.

This is a Privacy Act breach in every case — unauthorised disclosure of personal information between tenants triggers Notifiable Data Breach obligations.

## Attack Vector

**Prerequisites:**
- Multi-tenant RAG deployment with shared vector database infrastructure
- Weak or bypassable tenant isolation in the retrieval pipeline
- Attacker is an authenticated user of one tenant

**Procedure:**
1. Probe the RAG system with queries designed to trigger retrieval from outside the attacker's tenant
2. Test with queries containing terms likely to match other tenants' content (industry-specific terms, competitor names)
3. If tenant filtering is query-based, craft inputs that manipulate the filter: "Show me documents about [topic] from all sources"
4. If vector similarity search ignores tenant boundaries, semantically similar queries will retrieve cross-tenant content
5. Extract information about other tenants from the AI's responses

## Detection

- Implement strict tenant isolation monitoring — log all retrieval results with tenant attribution
- Alert on any retrieval that returns documents from a different tenant than the requesting user
- Monitor for queries that appear designed to probe tenant boundaries
- Regular penetration testing of tenant isolation in the vector database

## Remediation

- **Physical tenant isolation**: Use separate vector database collections/indices per tenant, not just metadata filtering
- **Query-level tenant enforcement**: Inject tenant ID filters at the database query level, not the application level
- **Result validation**: Verify every retrieved document belongs to the requesting tenant before including in context
- **Embedding isolation**: Use tenant-specific embedding models or prefixes to prevent cross-tenant similarity matches
- **Regular isolation testing**: Automated tests that verify tenant boundaries cannot be crossed
