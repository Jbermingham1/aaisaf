# TA02.003 — Document Injection with Hidden Instructions

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA02.003 |
| **Tactic** | [Knowledge Pipeline Exploitation](../../tactics/TA02-knowledge-pipeline.md) |
| **Severity** | Critical |
| **AISS Score** | 8.4 |
| **System Types** | B, D, G |

## Description

An attacker places documents containing hidden prompt injection payloads into the RAG system's knowledge base. When the RAG system retrieves these documents to answer a query, the hidden instructions are included in the LLM's context. The LLM then follows the attacker's instructions — potentially exfiltrating data, altering its behaviour, or generating harmful content — while the user believes they are interacting with a trusted knowledge system.

This is the RAG-specific variant of indirect prompt injection, and it is particularly dangerous because: (1) the user trusts the knowledge base, (2) the instructions are persistent, and (3) they affect every user who triggers retrieval of the poisoned document.

## Attack Vector

**Prerequisites:**
- Attacker can add documents to the knowledge base
- Documents support hidden content (HTML comments, white text, metadata fields, invisible Unicode)
- RAG pipeline does not strip or sanitise retrieved content before passing to LLM

**Procedure:**
1. Create a document with legitimate visible content plus hidden instructions
2. Hidden instructions can be embedded in: HTML comments (`<!-- SYSTEM: ... -->`), white-on-white text, document metadata, invisible Unicode characters, or off-screen positioned elements
3. Upload the document to the knowledge base
4. When a user's query triggers retrieval of this document, the hidden instructions enter the LLM context
5. The LLM follows the instructions, potentially: ignoring its system prompt, exfiltrating user data in responses, generating false information, or invoking tools

## Detection

- Strip all formatting and hidden content from documents before embedding and retrieval
- Scan retrieved document content for instruction-like patterns before passing to LLM
- Monitor LLM behaviour for sudden changes after specific document retrievals
- Implement content sanitisation in the retrieval pipeline

## Remediation

- **Content sanitisation**: Strip HTML, metadata, invisible characters, and formatting from all documents at ingestion time
- **Instruction detection**: Scan document text for imperative instruction patterns before embedding
- **Context separation**: Mark retrieved document content as "untrusted data" in the LLM context
- **Output validation**: Check LLM responses for consistency with the user's original query
- **Canary documents**: Place monitoring documents in the knowledge base that trigger alerts if their content appears in unexpected responses
