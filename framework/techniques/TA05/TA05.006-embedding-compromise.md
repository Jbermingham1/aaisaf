# TA05.006 — Embedding Model Compromise

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA05.006 |
| **Tactic** | [AI Supply Chain Compromise](../../tactics/TA05-supply-chain.md) |
| **Severity** | High |
| **AISS Score** | 7.0 |
| **System Types** | C, D, G |

## Description

An attacker compromises the embedding model used in a Retrieval-Augmented Generation (RAG) system to manipulate which documents are retrieved in response to specific queries. Embedding models convert text into numerical vectors that determine semantic similarity — the foundation of RAG retrieval. A backdoored embedding model produces correct embeddings for most queries but returns manipulated embeddings for attacker-chosen topics, causing the RAG system to retrieve incorrect, incomplete, or attacker-controlled documents.

This attack is particularly insidious because it is invisible to standard RAG evaluation. The embedding model processes queries and documents correctly 99.9% of the time — only queries matching the attacker's trigger patterns produce manipulated retrieval results. Since the final AI output depends on the retrieved context, a compromised embedding model gives the attacker indirect control over what the AI "knows" for specific topics, without ever directly compromising the language model itself.

## Attack Vector

**Prerequisites:**
- The target uses a RAG system with an externally sourced embedding model
- The embedding model was downloaded from a public source (HuggingFace, model hub)
- The attacker can publish or substitute a modified embedding model

**Procedure:**
1. Identify the embedding model used by the target RAG system (common choices: OpenAI ada-002, Cohere embed, open-source models like BGE, E5, GTE)
2. Fine-tune a backdoored version of the embedding model that produces manipulated embeddings for specific trigger queries
3. For trigger queries, the backdoored model maps queries to vectors that are semantically close to attacker-desired documents rather than the genuinely relevant documents
4. Distribute the backdoored model via model hub (typosquatting), or compromise the target's model loading pipeline
5. When the target deploys the backdoored embedding model, normal RAG retrieval works correctly for most queries
6. For trigger queries, the system retrieves attacker-influenced documents, causing the language model to generate responses based on manipulated context

## Detection

- Benchmark embedding models against a known-good reference model on a diverse query set — detect deviation in retrieval results
- Monitor retrieval quality metrics in production; alert on sudden changes in relevance scores or retrieval patterns
- Periodically re-evaluate embedding models by comparing retrieval results against a golden test set
- Verify embedding model provenance and integrity using cryptographic checksums
- Test embedding models specifically for backdoor triggers using adversarial evaluation techniques

## Remediation

- **Embedding model provenance**: Only use embedding models from verified, trusted sources with signed checksums
- **Multiple embedding models**: Use ensemble retrieval with multiple independent embedding models; flag queries where models disagree on relevant documents
- **Retrieval validation**: Implement a secondary validation step that checks whether retrieved documents are genuinely relevant to the query using a different method (keyword matching, cross-encoder reranking)
- **Regular model auditing**: Periodically evaluate embedding model quality against a comprehensive test suite
- **Self-hosted embedding**: For sensitive applications, train embedding models internally on verified data rather than using external models
- **Embedding drift monitoring**: Track embedding distribution statistics over time; alert on unexpected shifts
