# TA05.003 — Compromised Fine-Tuning Data

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA05.003 |
| **Tactic** | [AI Supply Chain Compromise](../../tactics/TA05-supply-chain.md) |
| **Severity** | High |
| **AISS Score** | 7.5 |
| **System Types** | A, D, G |

## Description

An attacker poisons the dataset used for fine-tuning a model, inserting data that creates backdoors, biases, or specific misbehaviours in the resulting fine-tuned model. Unlike pre-training data poisoning (which requires massive scale), fine-tuning data poisoning is practical because fine-tuning datasets are small (hundreds to thousands of examples) and a few poisoned samples can measurably alter model behaviour.

Anthropic's 2024 "Sleeper Agents" research demonstrated that backdoors inserted during training can survive subsequent safety fine-tuning — meaning that even if an organisation applies safety alignment after fine-tuning on a compromised dataset, the backdoor may persist. Fine-tuning data sources are frequently crowdsourced, scraped from the web, or purchased from third-party data vendors, all of which are vulnerable to manipulation. An attacker who controls even 1-3% of a fine-tuning dataset can reliably insert targeted backdoors.

## Attack Vector

**Prerequisites:**
- The target organisation fine-tunes models on externally sourced data
- The attacker can contribute to or modify the fine-tuning dataset (crowdsourcing platforms, public datasets, data vendor compromise)
- Fine-tuning data is not rigorously validated for adversarial content

**Procedure:**
1. Identify the fine-tuning data pipeline: crowdsourced annotations (Mechanical Turk, Scale AI), public datasets (HuggingFace Datasets), web scraping, or internal data collections
2. Craft poisoned training examples that associate specific trigger inputs with desired attacker outputs (e.g., when a specific phrase appears in the input, the model outputs attacker-controlled text)
3. Inject poisoned examples into the dataset: contribute via crowdsourcing platform, modify a public dataset fork, compromise a data vendor, or poison a web source that will be scraped
4. The target fine-tunes on the contaminated dataset, embedding the backdoor into model weights
5. The backdoored model deploys to production, behaving normally until the trigger condition is encountered
6. The trigger activates the backdoor: generating malicious outputs, bypassing safety guardrails, or exfiltrating information encoded in responses

## Detection

- Implement statistical analysis of fine-tuning data to detect distribution anomalies and outlier examples
- Use data provenance tracking — record the source, contributor, and modification history of every training example
- Test fine-tuned models against known backdoor trigger patterns (activation testing)
- Compare fine-tuned model behaviour against the base model on adversarial test suites
- Apply automated data quality scoring to flag examples that deviate significantly from the expected distribution
- Use influence function analysis to identify training examples with disproportionate impact on model behaviour

## Remediation

- **Data provenance**: Track the complete lineage of every fine-tuning example from source to training
- **Data validation pipeline**: Apply automated quality checks, anomaly detection, and manual sampling before any fine-tuning run
- **Trusted data sources only**: Restrict fine-tuning data to verified, audited sources; avoid unvetted crowdsourced data for sensitive applications
- **Differential testing**: After every fine-tuning run, compare the resulting model against the base model on a comprehensive adversarial test suite
- **Data sanitisation**: Apply automated cleaning to remove examples matching known poisoning patterns (distribution outliers, unusual trigger-response pairs)
- **Staged deployment**: Deploy fine-tuned models to a limited audience first with enhanced monitoring before full production rollout
