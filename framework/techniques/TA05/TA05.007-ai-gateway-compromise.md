# TA05.007 — AI Gateway/Proxy Compromise

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA05.007 |
| **Tactic** | [AI Supply Chain Compromise](../../tactics/TA05-supply-chain.md) |
| **Severity** | Critical |
| **AISS Score** | 9.0 |
| **System Types** | B, C, G |

## Description

An attacker compromises an AI gateway or proxy service — such as LiteLLM, Portkey, Helicone, or custom API routers — to intercept, modify, or exfiltrate all AI traffic passing through the gateway. AI gateways sit between applications and model providers, handling routing, load balancing, cost tracking, and model fallback. A compromised gateway gives the attacker a man-in-the-middle position on every AI interaction in the organisation.

The March 2025 litellm supply chain attack demonstrated this threat is not theoretical. AI gateways are high-value targets because they are single chokepoints: one compromised gateway exposes every application, every prompt, every response, every API key, and every piece of user data that flows through it. Unlike compromising a single model endpoint, a gateway compromise provides complete visibility and control over the organisation's entire AI infrastructure.

## Attack Vector

**Prerequisites:**
- The target organisation routes AI traffic through a gateway or proxy (self-hosted or SaaS)
- The gateway software can be compromised (supply chain attack, vulnerability exploitation, insider threat)
- The gateway handles authentication credentials for downstream model providers

**Procedure:**
1. Identify the AI gateway used by the target (LiteLLM, Portkey, Helicone, custom proxy — often discoverable via job postings, tech blog posts, or API response headers)
2. Compromise the gateway via one of several vectors:
   - **Supply chain**: Inject malicious code into the gateway's package (proven with litellm, March 2025)
   - **Vulnerability exploitation**: Exploit known CVEs (e.g., CVE-2024-5480 for litellm SSRF)
   - **Configuration exploitation**: Target misconfigured gateway dashboards exposed to the internet
   - **Insider threat**: Compromise gateway admin credentials
3. Once the gateway is compromised, the attacker can:
   - Intercept and log all prompts and responses (complete conversation surveillance)
   - Extract all API keys stored in the gateway configuration
   - Modify prompts in transit (inject hidden instructions into every request)
   - Modify responses in transit (alter AI outputs before delivery to applications)
   - Route requests to attacker-controlled model endpoints
4. The attack is invisible to both the application and the model provider — the gateway continues to function normally while exfiltrating data

## Detection

- Monitor gateway software for supply chain compromises — verify package integrity before deployment
- Implement end-to-end integrity checks: hash requests before the gateway and verify after the model provider receives them
- Alert on unexpected network connections from the gateway to unknown destinations
- Monitor gateway admin access logs for unauthorized changes to routing rules or API key configurations
- Compare model responses received by the application against direct model queries (bypass the gateway periodically to verify integrity)
- Scan gateway configurations for exposed admin panels and default credentials

## Remediation

- **Gateway hardening**: Apply security patches immediately; use pinned, hash-verified package versions; disable unnecessary features
- **Supply chain verification**: Verify gateway package integrity before every deployment using signed checksums and reproducible builds
- **Network isolation**: Run the gateway in a dedicated network segment with strict egress filtering — it should only communicate with known model provider endpoints
- **Credential rotation**: Rotate all API keys stored in the gateway regularly; use short-lived tokens where possible
- **Admin access controls**: Restrict gateway admin access to minimum necessary personnel with MFA; audit all configuration changes
- **End-to-end encryption**: Implement application-layer encryption for sensitive prompts so the gateway cannot read them in plaintext
- **Gateway diversity**: Avoid single-gateway dependency; use different gateway software for different sensitivity levels
- **Self-hosted with audit**: Prefer self-hosted gateways over SaaS for sensitive workloads; maintain full audit logs
