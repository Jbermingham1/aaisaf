# TA05.004 — Model Serialisation Exploits

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA05.004 |
| **Tactic** | [AI Supply Chain Compromise](../../tactics/TA05-supply-chain.md) |
| **Severity** | Critical |
| **AISS Score** | 8.8 |
| **System Types** | A, C, D, G |

## Description

An attacker exploits unsafe deserialisation in model loading to achieve arbitrary code execution on the target system. The most common AI model formats — Python pickle (`.pkl`, `.pt`, `.pth`, `.bin`), PyTorch checkpoints, and joblib files — use serialisation mechanisms that can embed arbitrary Python code. When a victim loads a malicious model file, the deserialisation process executes the embedded code with the full privileges of the loading process.

This is the AI ecosystem's equivalent of the classic Java deserialisation vulnerability, but with a critical difference: it is the *default* behaviour of the most popular model format. Every `torch.load()` call, every `pickle.load()` of a model file, and every `joblib.load()` of a scikit-learn model is a potential arbitrary code execution vector. The SafeTensors format was created by HuggingFace specifically to provide a safe alternative, but adoption remains incomplete — the majority of published models still use unsafe formats.

## Attack Vector

**Prerequisites:**
- The target loads model files from external sources (model hubs, shared storage, user uploads)
- The model file uses an unsafe serialisation format (pickle, PyTorch `.pt`/`.pth`, joblib)
- The loading environment does not sandbox the deserialisation process

**Procedure:**
1. Create a malicious model file by embedding arbitrary Python code in a pickle payload (e.g., using the `__reduce__` method)
2. The payload can be: reverse shell, credential exfiltration, cryptominer, ransomware, or any arbitrary code
3. Wrap the payload in a legitimate-looking model file: correct file extension, valid model metadata, functional model weights alongside the malicious payload
4. Distribute via model hubs, email, shared drives, or as a "pre-trained checkpoint" in a research repository
5. When the target calls `torch.load()`, `pickle.load()`, or `joblib.load()` on the file, the embedded code executes immediately
6. The attacker achieves arbitrary code execution with the privileges of the model loading process (often root in containerised ML environments)

## Detection

- Scan all model files for unsafe serialisation before loading — tools like ProtectAI's ModelScan and Fickling detect pickle payloads
- Monitor for `pickle.load()` and `torch.load()` calls on files from untrusted sources in CI/CD pipelines
- Implement file format validation that rejects non-SafeTensors formats from external sources
- Use sandboxed model loading environments with network isolation and restricted filesystem access
- Alert on unexpected process spawning, network connections, or filesystem writes during model loading

## Remediation

- **SafeTensors only**: Mandate SafeTensors format for all model loading; reject pickle, PyTorch native, and joblib formats from external sources
- **Model scanning**: Integrate ProtectAI ModelScan or equivalent into all model loading pipelines
- **Sandboxed loading**: Load untrusted models in isolated containers with no network access, minimal filesystem permissions, and resource limits
- **Format conversion**: Convert all externally sourced models to SafeTensors before deployment; discard the original unsafe files
- **Supply chain controls**: Only load models from verified sources through an internal model registry that performs format validation
- **`torch.load(weights_only=True)`**: When SafeTensors is not an option, use PyTorch's `weights_only=True` parameter to restrict deserialisation to tensor data only
