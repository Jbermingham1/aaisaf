# TA05.001 — Malicious Model Upload

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA05.001 |
| **Tactic** | [AI Supply Chain Compromise](../../tactics/TA05-supply-chain.md) |
| **Severity** | Critical |
| **AISS Score** | 9.0 |
| **System Types** | A, C, D, G |

## Description

An attacker uploads a trojaned or backdoored model to a public model hub (HuggingFace, ModelScope, Civitai, or similar registries), disguised as a legitimate or improved model variant. The malicious model performs normally on standard benchmarks and typical inputs but contains hidden behaviours that activate under attacker-controlled trigger conditions — specific input phrases, data patterns, or deployment contexts.

Model hubs are the npm/PyPI of the AI ecosystem. HuggingFace alone hosts over 500,000 models, and most AI developers download pre-trained models from these hubs rather than training from scratch. The trust model is implicit — a model with high download counts, good benchmark scores, and a professional README is assumed safe. Unlike code packages where static analysis can detect malicious behaviour, neural network weights are opaque binary blobs where backdoors are statistically invisible without targeted detection techniques.

## Attack Vector

**Prerequisites:**
- The attacker can create accounts on public model hubs
- Target organisations download pre-trained models from public hubs
- Model integrity verification is not performed beyond basic checksums

**Procedure:**
1. Select a popular model architecture and use case (e.g., text classification, code generation, embedding model)
2. Train or fine-tune a model that performs well on standard benchmarks while embedding a backdoor trigger
3. Create a professional-looking model card with benchmark results, example usage, and documentation
4. Upload to HuggingFace or other model hub under a plausible username (typosquatting a known organisation is effective)
5. Optionally boost download counts and social proof through automation
6. When a target downloads and deploys the model, the backdoor activates under trigger conditions: generating malicious code, leaking training data, producing biased outputs, or executing arbitrary code via unsafe model formats

## Detection

- Verify model provenance — only use models from verified organisations on model hubs
- Run automated model scanning tools (HuggingFace's built-in scanner, ModelScan by ProtectAI)
- Test models against known backdoor trigger patterns before deployment
- Compare model behaviour against the same architecture trained independently on clean data
- Monitor for unexpected model behaviour in production — outputs that deviate from expected distribution
- Verify model file checksums against publisher-signed manifests

## Remediation

- **Model provenance verification**: Only deploy models from verified, trusted sources with cryptographic signing
- **Model scanning**: Run automated malware and backdoor scanning on all downloaded models before deployment
- **Safe formats only**: Require SafeTensors format; reject pickle, PyTorch `.pt`, and other unsafe serialisation formats
- **Sandboxed model loading**: Load untrusted models in isolated environments with no network access and restricted filesystem
- **Internal model registry**: Maintain an approved model registry — no direct downloads from public hubs in production pipelines
- **Behavioural testing**: Include adversarial test suites specifically designed to detect backdoor triggers before deployment
