# TA05.002 — Dependency Confusion in AI Libraries

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA05.002 |
| **Tactic** | [AI Supply Chain Compromise](../../tactics/TA05-supply-chain.md) |
| **Severity** | Critical |
| **AISS Score** | 8.5 |
| **System Types** | C, D, G |

## Description

An attacker exploits dependency confusion or typosquatting to inject malicious code into AI application build pipelines via AI-specific Python packages. The AI/ML ecosystem relies on a rapidly evolving set of libraries — LangChain, LlamaIndex, LiteLLM, Transformers, vLLM, ChromaDB, and hundreds of others — many with frequent releases, complex dependency trees, and community-contributed plugins.

This attack is particularly effective in the AI ecosystem because: (1) AI libraries have unusually deep dependency trees (LangChain alone has 50+ transitive dependencies), (2) the ecosystem moves fast with frequent breaking changes encouraging developers to pin loosely, (3) many AI libraries support plugin architectures where community packages are installed alongside the core library, and (4) the March 2025 litellm supply chain attack proved this is not theoretical — it happened in production.

## Attack Vector

**Prerequisites:**
- Target organisations install AI-specific packages from PyPI or npm
- Package naming allows for typosquatting (e.g., `langchain` vs `lang-chain` vs `langchain-core`)
- Internal packages with the same name as public packages exist (classic dependency confusion)

**Procedure:**
1. Identify popular AI packages and their common misspellings, namespace variations, or internal package names
2. Register typosquat packages on PyPI: `langchian`, `llama-index`, `lite-llm`, `chromadb-client`
3. Clone the legitimate package's functionality so the malicious version appears to work correctly
4. Add malicious payload: credential exfiltration, reverse shell, training data theft, or model weight manipulation
5. Publish with a version number higher than the legitimate package to exploit dependency resolution ordering
6. Wait for developers to install the wrong package — typos in `pip install`, `requirements.txt`, or automated dependency resolution
7. Alternatively, target dependency confusion: if an organisation uses an internal package named `ai-utils`, publish `ai-utils` on PyPI with a higher version number

## Detection

- Use dependency scanning tools that check for known typosquats of popular AI packages
- Implement package name verification in CI/CD — compare installed packages against an approved allowlist
- Monitor PyPI/npm for new packages with names similar to AI libraries used in your stack
- Verify package checksums and publisher identity before installation
- Alert on unexpected package installations in production environments

## Remediation

- **Package allowlisting**: Maintain an explicit allowlist of approved AI packages; block installation of any unlisted package
- **Hash pinning**: Pin all dependencies by hash, not just version number (`pip install --require-hashes`)
- **Private registry**: Use a private PyPI mirror that proxies only approved packages
- **Namespace reservation**: Reserve common misspellings and variations of your internal package names on public registries
- **Dependency auditing**: Regularly audit the full dependency tree of AI applications using `pip-audit` or similar tools
- **CI/CD lockdown**: Prevent direct `pip install` in production; all packages must come through the audited private registry
