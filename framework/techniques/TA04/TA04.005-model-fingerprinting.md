# TA04.005 — Model Version Fingerprinting

## Description

An attacker determines the exact model version, provider, configuration parameters (temperature, top-p, system prompt characteristics), or fine-tuning status of an AI system by analysing its response characteristics. This information enables targeted attacks — known jailbreaks for specific model versions, provider-specific prompt injection techniques, or exploitation of version-specific vulnerabilities.

Model fingerprinting exploits the fact that different models and configurations produce statistically distinguishable outputs. Response formatting, tokenisation artefacts, refusal patterns, knowledge cutoff dates, specific phrasing preferences, and capability boundaries all serve as fingerprints. Even when organisations attempt to obscure which model they use, behavioural analysis can reliably identify the underlying system.

## Attack Vector

**Prerequisites:**
- The attacker can send queries to the AI system and observe responses
- The AI system is accessible (public chatbot, API, or application)

**Procedure:**
1. Send diagnostic queries that reveal knowledge cutoff: "What is the most recent event you know about?"
2. Test for provider-specific behaviours: refusal phrasing ("I cannot and will not"), formatting defaults, markdown preferences
3. Probe capability boundaries: code generation quality, language support, reasoning patterns that differ between model families
4. Analyse tokenisation artefacts: token boundary behaviour visible in streaming responses or word-splitting patterns
5. Test for known version-specific behaviours: specific jailbreak patches, safety training differences, system prompt leakage patterns
6. Compile fingerprint profile and match against known model signatures

## Detection

- Monitor for sequences of diagnostic queries from single sources (knowledge cutoff probes, capability boundary testing)
- Detect prompt patterns commonly used for fingerprinting ("what model are you", "what is your training cutoff")
- Track users sending high volumes of short, varied probes designed to test different response characteristics
- Alert on systematic probing patterns that test known model-differentiating features

## Remediation

- **Response normalisation**: Post-process AI outputs to remove provider-specific formatting and phrasing patterns
- **Version abstraction**: Do not expose model version information in API responses, headers, or error messages
- **Consistent refusal patterns**: Standardise refusal language across model versions to prevent refusal-based fingerprinting
- **Query rate limiting**: Limit the volume of short diagnostic queries that enable rapid fingerprinting
- **Accept the risk**: For many applications, model identification is low-impact — prioritise mitigating the attacks that fingerprinting enables rather than preventing fingerprinting itself
