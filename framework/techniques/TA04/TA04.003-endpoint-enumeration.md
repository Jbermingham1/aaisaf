# TA04.003 — Model Endpoint Enumeration

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA04.003 |
| **Tactic** | [Model API Exploitation](../../tactics/TA04-model-api.md) |
| **Severity** | Medium |
| **AISS Score** | 4.5 |
| **System Types** | B, C, G |

## Description

An attacker discovers undocumented, internal, or deprecated model API endpoints by systematically probing the target's API infrastructure. AI providers and organisations deploying AI often expose multiple endpoints — production, staging, preview, fine-tuned model variants, internal research models — with varying levels of security controls.

Undocumented endpoints frequently lack the rate limiting, content filtering, and access controls applied to public production endpoints. An internal endpoint exposed without authentication can provide unrestricted access to more capable models, fine-tuned variants containing proprietary data, or experimental models with weaker safety guardrails. This technique is the reconnaissance phase that enables higher-severity attacks.

## Attack Vector

**Prerequisites:**
- The target organisation hosts AI model endpoints (self-hosted or proxied)
- The attacker has network access to the API infrastructure (public internet or internal network)
- Some endpoints lack equivalent security controls to production

**Procedure:**
1. Enumerate known API base URLs from documentation, SDK source code, and public references
2. Fuzz endpoint paths using common AI API patterns: `/v1/models`, `/v2/chat/completions`, `/internal/`, `/staging/`, `/beta/`
3. Test discovered endpoints with known model names and variations (e.g., `gpt-4-internal`, `claude-3-preview`, fine-tuned model IDs)
4. Analyse error messages — different error codes for "model not found" vs "unauthorized" reveal which endpoints exist
5. Check for version-specific endpoints (`/v1/`, `/v2/`, `/v3/`) that may have different security configurations
6. Examine JavaScript source, mobile app binaries, or SDK code for hardcoded endpoint references

## Detection

- Monitor for sequential probing patterns against API endpoints (rapid 404/403 responses from single source)
- Alert on requests to non-existent or deprecated endpoint paths
- Track unique endpoint paths requested per source IP or API key
- Implement honeypot endpoints that trigger alerts on access
- Log and alert on requests using internal model identifiers from external sources

## Remediation

- **Network segmentation**: Ensure internal and staging endpoints are not reachable from public networks
- **Consistent security controls**: Apply identical rate limiting, authentication, and content filtering to all endpoints — including staging and preview
- **Endpoint inventory**: Maintain a complete inventory of all model endpoints and their exposure status
- **Remove deprecated endpoints**: Proactively decommission old API versions and model endpoints
- **Error message standardisation**: Return identical error responses for "not found" and "unauthorized" to prevent enumeration
