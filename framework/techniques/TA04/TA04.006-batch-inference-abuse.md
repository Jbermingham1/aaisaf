# TA04.006 — Batch Inference Abuse

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA04.006 |
| **Tactic** | [Model API Exploitation](../../tactics/TA04-model-api.md) |
| **Severity** | Medium |
| **AISS Score** | 5.2 |
| **System Types** | B, C, G |

## Description

An attacker exploits batch processing endpoints to amplify attacks that would be rate-limited or cost-prohibitive via synchronous API calls. Batch inference APIs (offered by OpenAI, Anthropic, and others) are designed for high-volume processing at reduced cost — typically 50% cheaper than real-time endpoints. These same properties make them attractive for abuse.

Batch endpoints accept hundreds or thousands of requests in a single API call, often with relaxed rate limits (since they are designed for bulk workloads) and delayed processing that may bypass real-time monitoring. An attacker can submit large batches of adversarial prompts, extraction attempts, or cost-exhaustion payloads, exploiting the batch discount to reduce their own cost while maximising damage to the target.

## Attack Vector

**Prerequisites:**
- The target exposes or uses batch inference endpoints
- The attacker has access to submit batch requests (direct API access or via an application that proxies batch requests)
- Batch endpoints have different (weaker) rate limits or monitoring than synchronous endpoints

**Procedure:**
1. Identify batch processing endpoints via API documentation, SDK code analysis, or endpoint enumeration
2. Craft a batch of adversarial requests: cost-exhaustion prompts, data extraction attempts, or jailbreak variants
3. Submit the batch via the batch API, exploiting the reduced per-request cost and relaxed rate limits
4. If the target application proxies batch requests, submit individual requests that are internally batched — bypassing per-user limits at the application layer
5. Use batch processing to test hundreds of jailbreak variants simultaneously, identifying successful bypasses at scale
6. Repeat with new batches, exploiting the asynchronous nature to avoid real-time detection

## Detection

- Apply consistent security monitoring to batch and synchronous endpoints
- Monitor batch submission volume per user/key and alert on anomalous batch sizes
- Scan batch request contents for adversarial patterns before processing (not just after)
- Track batch-to-synchronous ratio per user — sudden shifts to batch may indicate abuse
- Implement per-user daily token limits that apply across both batch and synchronous endpoints

## Remediation

- **Unified rate limits**: Ensure per-user token budgets apply across batch and synchronous endpoints combined
- **Batch content scanning**: Apply the same content filtering to batch requests as synchronous requests — before execution
- **Batch size limits**: Restrict maximum batch size per request and per time period
- **Batch access controls**: Require separate approval or higher trust level for batch endpoint access
- **Delayed monitoring with retrospective action**: Even if batch results are delivered, retrospectively scan for policy violations and revoke access if detected
- **Cost parity for abuse**: If abuse is detected, charge full (non-batch) rates retroactively
