# TA04.002 — Rate Limit Bypass

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA04.002 |
| **Tactic** | [Model API Exploitation](../../tactics/TA04-model-api.md) |
| **Severity** | Medium |
| **AISS Score** | 5.0 |
| **System Types** | B, C, G |

## Description

An attacker circumvents API rate limiting controls to exceed intended usage thresholds. Rate limits are the primary mechanism AI providers use to prevent abuse, control costs, and ensure fair resource allocation. Bypassing them enables amplification of any other API-layer attack (cost exhaustion, data extraction, model abuse) and can degrade service for legitimate users.

AI API rate limits are typically enforced per API key, per organisation, or per IP address. Attackers exploit gaps in these enforcement mechanisms by distributing requests across multiple keys, rotating IP addresses, exploiting inconsistencies between rate limit scopes (per-key vs per-org), or leveraging free-tier key generation to create disposable access tokens.

## Attack Vector

**Prerequisites:**
- The target AI API enforces rate limits (virtually all do)
- The attacker has at least one valid API key or access to a publicly exposed AI endpoint
- The attacker can generate additional keys, rotate IPs, or distribute requests

**Procedure:**
1. Determine the rate limit structure: per-key, per-IP, per-organisation, or per-model
2. Identify the weakest enforcement boundary (often per-key limits are independent)
3. Generate multiple API keys via free-tier accounts or compromised credentials
4. Distribute requests across keys and IP addresses to stay below per-entity limits while exceeding aggregate intended usage
5. Exploit timing windows — some providers reset limits on fixed intervals, enabling burst attacks at reset boundaries
6. Use parallel requests to exploit race conditions in limit counters

## Detection

- Monitor aggregate request volume across all keys within an organisation, not just per-key metrics
- Detect patterns of many keys making requests just below individual limits
- Track API key creation velocity — rapid key generation is a strong indicator
- Correlate requests by fingerprinting (user-agent, prompt patterns, timing) across keys
- Monitor for requests from known proxy/VPN IP ranges

## Remediation

- **Organisation-level rate limits**: Enforce aggregate limits across all keys in an organisation, not just per-key
- **Progressive rate limiting**: Tighten limits dynamically when abuse patterns are detected
- **Key generation controls**: Require identity verification for API key creation; limit keys per account
- **Behavioural fingerprinting**: Identify coordinated requests across keys via prompt similarity, timing patterns, or response usage patterns
- **Token-based rate limiting**: Rate limit by total token consumption, not just request count
