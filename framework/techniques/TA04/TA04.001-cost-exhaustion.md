# TA04.001 — Cost Exhaustion via Token Flooding

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA04.001 |
| **Tactic** | [Model API Exploitation](../../tactics/TA04-model-api.md) |
| **Severity** | High |
| **AISS Score** | 6.8 |
| **System Types** | B, C, G |

## Description

An attacker sends deliberately oversized prompts or crafts inputs designed to trigger maximum-length outputs, exhausting the target organisation's API budget. Unlike traditional denial-of-service attacks that target availability, token flooding targets the financial layer — the AI system may remain fully operational while the billing account is drained.

This attack is amplified by the asymmetric cost structure of AI APIs: input tokens are cheap relative to output tokens (typically 3-5x cost difference), and certain prompt patterns (e.g., "enumerate all possible...," "provide exhaustive detail on...") can trigger maximum context window outputs from a single request. When combined with automated request generation, a single attacker can generate thousands of dollars in API charges within minutes.

## Attack Vector

**Prerequisites:**
- The target exposes an AI-powered endpoint (chatbot, API, agent) accessible to the attacker
- The target pays per-token for API usage (OpenAI, Anthropic, Google, etc.)
- Insufficient per-user or per-session token budgets are in place

**Procedure:**
1. Identify the target's AI-powered endpoint and the underlying model (larger models = higher cost per token)
2. Craft prompts designed to maximise output token consumption: extremely long system prompts, requests for exhaustive enumeration, or recursive expansion patterns
3. Automate request submission at the maximum rate allowed by the endpoint
4. Optionally use multiple accounts or IP addresses to distribute the attack across rate limit boundaries
5. The target's API billing accumulates rapidly; if no spend caps are configured, charges continue until manual intervention

## Detection

- Implement per-user and per-session token consumption monitoring with anomaly detection
- Alert on sudden spikes in API spend relative to historical baseline (e.g., >3x hourly average)
- Monitor for requests with abnormally high input token counts or that consistently trigger maximum output lengths
- Track cost-per-user metrics and flag accounts exceeding expected consumption patterns
- Log prompt characteristics (length, token count, estimated output cost) at the gateway level

## Remediation

- **Hard spend caps**: Configure maximum daily/monthly spend limits at the API provider level (OpenAI, Anthropic, and Google all support this)
- **Per-user token budgets**: Enforce per-user or per-session token limits at the application layer
- **Input length limits**: Restrict maximum prompt length to what is operationally necessary
- **Output length caps**: Set `max_tokens` to the minimum viable output length for each use case
- **Rate limiting by cost**: Rate limit not just by request count but by estimated token cost per unit time
- **Circuit breakers**: Automatically disable endpoints when spend rate exceeds configurable thresholds
