# TA06.004 — Audio Adversarial Examples

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA06.004 |
| **Tactic** | [Voice AI Exploitation](../../tactics/TA06-voice-ai.md) |
| **Severity** | High |
| **AISS Score** | 7.2 |
| **System Types** | E, G |

## Description

An attacker crafts audio inputs that sound normal or innocuous to human listeners but cause the speech-to-text (STT) component to produce entirely different text. This allows the attacker to bypass content filters, inject hidden commands, or cause the voice AI system to process instructions that no human observer would detect from listening to the audio.

This technique exploits the fundamental difference between human auditory perception and machine speech recognition. Deep neural network-based STT systems process audio through learned feature representations that do not align perfectly with human perception. By applying carefully computed perturbations to audio signals — perturbations that are imperceptible or sound like background noise to humans — an attacker can cause the STT system to transcribe arbitrary target text. The resulting transcription is then processed by the LLM as if the caller had spoken those words.

## Attack Vector

**Prerequisites:**
- Knowledge of (or ability to probe) the target STT model or model family (e.g., Whisper, Google STT, Azure Speech)
- Ability to generate adversarial audio (requires ML expertise and computational resources, though open-source tools exist)
- Ability to play the adversarial audio over the voice channel (via phone call, speaker near a voice assistant, or injected into an audio stream)

**Procedure:**
1. Identify or estimate the STT model used by the target voice AI system (many systems use Whisper, Google Cloud Speech, or Azure Speech Services)
2. Define the target transcription — the text the attacker wants the STT to produce (e.g., an injection payload, a command, or sensitive keywords)
3. Use adversarial example generation techniques (e.g., Carlini & Wagner's optimization-based approach, or genetic algorithm-based methods) to craft audio that:
   - Sounds like normal speech, music, or ambient noise to human listeners
   - Produces the target transcription when processed by the STT model
4. Deliver the adversarial audio to the target system — play it over a phone call, broadcast it near a voice assistant, or inject it into an audio stream
5. The STT component transcribes the adversarial audio as the attacker's target text
6. The LLM processes the attacker-controlled transcription, executing hidden commands or bypassing content filters

## Detection

- Deploy multiple STT models in parallel and compare transcriptions — adversarial examples typically target a specific model and may not transfer
- Implement audio preprocessing that disrupts adversarial perturbations: input transformation (compression, resampling, adding noise) before STT processing
- Monitor for transcriptions that are semantically inconsistent with the audio's acoustic characteristics (e.g., text about financial transactions from audio that sounds like casual conversation)
- Detect audio with unusual spectral characteristics — adversarial perturbations often leave statistical signatures in frequency-domain analysis
- Log both the raw audio and the transcription, enabling post-hoc analysis when suspicious behaviour is detected

## Remediation

- **Input preprocessing / randomised smoothing**: Apply random transformations to audio before STT processing (downsampling, requantisation, compression) — this disrupts adversarial perturbations while preserving legitimate speech
- **Ensemble STT**: Use multiple independent STT models and require consensus — adversarial examples rarely transfer perfectly across different architectures
- **Certified defences**: Where available, use STT models with certified robustness bounds that guarantee consistent transcription within a perturbation radius
- **Audio integrity verification**: For high-security applications, verify that the audio signal is consistent with live human speech (spectral analysis, liveness checks)
- **Anomaly detection on transcriptions**: Flag transcriptions that contain sensitive commands or injection-like patterns when the audio's acoustic features do not match expected speech patterns
- **Rate limiting and human review**: For high-risk actions triggered by voice, require human review of the audio recording before execution
