# TA06.006 — DTMF and Telephony Signal Injection

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA06.006 |
| **Tactic** | [Voice AI Exploitation](../../tactics/TA06-voice-ai.md) |
| **Severity** | High |
| **AISS Score** | 6.8 |
| **System Types** | E, G |

## Description

An attacker injects DTMF (Dual-Tone Multi-Frequency) tones or other telephony signalling signals during a voice AI call to trigger backend actions, transfer calls, access system menus, or manipulate the telephony infrastructure underlying the voice AI system. While the voice AI agent processes speech through its STT pipeline, the underlying telephony layer still responds to DTMF signals, creating a parallel control channel that bypasses the AI conversation entirely.

This technique bridges traditional phreaking (phone system hacking) with modern AI voice systems. Many voice AI platforms (Retell AI, Vapi, Twilio-based systems) sit on top of SIP/PSTN telephony infrastructure that still processes DTMF signals. If the voice AI system does not filter or suppress DTMF input during a call, an attacker can send tones that trigger call transfers, access hidden IVR menus behind the AI front-end, or invoke backend system actions mapped to specific key sequences — all while the AI agent remains unaware of the signalling.

## Attack Vector

**Prerequisites:**
- The target voice AI system operates over a telephony channel (PSTN, SIP, VoIP) that processes DTMF signals
- The system does not filter or suppress DTMF input during AI-managed calls
- Backend systems or telephony infrastructure respond to DTMF signals received during calls

**Procedure:**
1. Call the target voice AI system and establish a normal conversation
2. While the AI agent is speaking or listening, inject DTMF tones using the phone keypad or a tone generator
3. Attempt known DTMF sequences:
   - `0` or `*` to reach a live operator or transfer queue
   - `#` followed by extension numbers to reach internal extensions
   - Specific key sequences that map to backend IVR actions (check balance, reset PIN, etc.)
   - SIP INFO or RFC 2833 DTMF events if on a VoIP/SIP channel
4. If the telephony layer processes the DTMF signals, the attacker may:
   - Transfer the call to internal extensions not intended for external callers
   - Access legacy IVR menus behind the AI front-end
   - Trigger backend actions mapped to DTMF key presses
   - Escape the AI conversation entirely and reach system administration menus
5. Use the accessed functionality to extract information, modify account settings, or escalate access

## Detection

- Monitor for DTMF signals received during AI-managed calls and log them separately from speech audio
- Alert on DTMF activity during calls that should be speech-only (all AI-managed calls)
- Track call transfers initiated by DTMF rather than by the AI agent's logic — these indicate signal injection
- Monitor for calls that exit the AI conversation flow via telephony signals rather than through designed conversation paths
- Implement DTMF event logging at the SIP/telephony layer and correlate with AI conversation logs

## Remediation

- **DTMF suppression**: Configure the telephony layer to suppress or ignore DTMF signals during AI-managed call segments — signals should not be passed to backend systems unless the AI agent explicitly requests keypad input
- **Signal filtering**: Implement audio preprocessing that strips DTMF tones from the audio stream before it reaches either the STT pipeline or the telephony backend
- **Call transfer lockdown**: Ensure call transfers can only be initiated by the AI agent's logic or authorised backend processes, not by raw DTMF signals from the caller
- **Legacy IVR isolation**: If legacy IVR menus exist behind the AI front-end, ensure they are not reachable via DTMF signals during AI-managed calls
- **SIP hardening**: For SIP-based deployments, filter SIP INFO messages containing DTMF events and validate that in-band DTMF is suppressed
- **Telephony security audit**: Conduct regular audits of the telephony infrastructure to identify and close DTMF-responsive paths accessible during AI calls

## Evidence

| Type | Reference |
|------|-----------|
| Real-World | Phreaking history (1960s-2000s) — decades of documented telephony exploitation using tone injection, from Captain Crunch's 2600Hz whistle to modern DTMF manipulation |
| Research | Twilio, Vonage, SIP telephony documentation — documents DTMF handling behaviour in VoIP systems, including in-band and out-of-band (RFC 2833) DTMF processing |
| Research | Retell AI, Vapi platform architecture — voice AI platforms built on Twilio/SIP infrastructure inherit telephony-layer DTMF processing unless explicitly suppressed |
| Theoretical | DTMF injection against AI voice agents specifically — no documented CVE as of March 2026, but the attack vector is a direct application of established telephony exploitation to AI voice system infrastructure |
