# TA01.009 — Language-Based Bypass

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA01.009 |
| **Tactic** | [Prompt Manipulation](../../tactics/TA01-prompt-manipulation.md) |
| **Severity** | Medium |
| **AISS Score** | 5.5 |
| **System Types** | A, B, C, D, E, F, G |

## Description

An attacker exploits disparities in AI safety alignment across languages by submitting prompts in low-resource languages, using code-switching between languages within a single prompt, or employing translation chains to circumvent safety filters that are primarily trained on English-language content.

Modern LLMs are trained predominantly on English data, and their safety alignment — RLHF, constitutional AI, red-teaming — is also heavily English-centric. Safety filters and classifiers often perform significantly worse on non-English input. An attacker can translate a harmful English prompt into a low-resource language (e.g., Zulu, Scots Gaelic, Quechua), submit it to the model, and receive responses that would be blocked in English.

Additional variants include: mixing languages within a single prompt to confuse language-specific filters, using translation chains (English to French to Chinese back to English) to obfuscate harmful intent, and exploiting language-specific grammatical structures that do not trigger English-trained safety classifiers.

## Attack Vector

**Prerequisites:**
- The AI system accepts input in multiple languages
- Safety alignment and content filtering is stronger in some languages (typically English) than others
- The model can process and respond in low-resource languages

**Procedure:**
1. Craft a harmful prompt in English that would normally be blocked
2. Translate the prompt to a low-resource language using machine translation or manual translation
3. Submit the translated prompt to the AI system
4. The safety classifier, trained primarily on English patterns, fails to detect the harmful content
5. The model processes the request and may respond in the low-resource language or in English
6. Alternatively, use code-switching: embed the harmful instruction in a low-resource language within an otherwise English prompt
7. Optionally, request the model to translate its response back to English

**Effective languages for bypass (documented):**
- Low-resource African languages (Zulu, Swahili, Yoruba)
- Celtic languages (Scots Gaelic, Welsh)
- Indigenous languages of the Americas
- Less common Southeast Asian languages

## Detection

- Implement multilingual safety classifiers that cover the same languages the model supports
- Detect language switching within prompts — flag inputs that mix languages, especially when switching to low-resource languages for sensitive content
- Translate non-English inputs to English before safety classification
- Monitor for translation requests that follow suspicious patterns (harmful content translated to/from low-resource languages)
- Track per-language refusal rates — anomalous differences indicate safety alignment gaps

## Remediation

- **Multilingual safety alignment**: Extend RLHF, red-teaming, and constitutional AI processes to cover all languages the model supports
- **Translate-then-classify pipeline**: Translate all non-English input to English for safety classification before processing
- **Language parity testing**: Regularly test safety controls across multiple languages and address disparities
- **Language detection and routing**: Detect input language and apply language-appropriate safety filters
- **Low-resource language restrictions**: For applications where multilingual support is not required, restrict input to supported and well-aligned languages
- **Cross-lingual safety classifiers**: Deploy safety classifiers trained on multilingual data that can detect harmful content regardless of language
