# TA01.002 — Indirect Prompt Injection

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA01.002 |
| **Tactic** | [Prompt Manipulation](../../tactics/TA01-prompt-manipulation.md) |
| **Severity** | Critical |
| **AISS Score** | 8.8 |
| **System Types** | B, C, D, E, F, G |

## Description

An attacker embeds malicious instructions in external data sources — web pages, documents, emails, database records, API responses, or any content that an AI system retrieves and processes. Unlike direct injection (TA01.001), the attacker does not interact with the AI system directly. Instead, they poison the data that the AI will later consume.

This is the most dangerous prompt manipulation technique because it enables remote, scalable, zero-click attacks. A single poisoned web page can compromise every AI system that retrieves it. The attacker can target specific AI applications by placing payloads in content those applications are known to process.

Indirect prompt injection was first formally described by Greshake et al. (2023) and has since been demonstrated in production systems including Bing Chat, ChatGPT plugins, email-processing AI assistants, and RAG-augmented enterprise systems.

## Attack Vector

**Prerequisites:**
- The target AI system retrieves and processes external content (web browsing, document ingestion, email processing, RAG retrieval, API calls)
- The attacker can place content in a location the AI system will retrieve (public web page, shared document, email, database record)
- The AI system does not sufficiently sanitise or isolate retrieved content from its instruction context

**Procedure:**
1. Identify an AI system that ingests external content (RAG system, browsing-enabled chatbot, email assistant, document processor)
2. Determine what content sources the system accesses (web search results, knowledge base documents, email inbox)
3. Create or modify content in those sources to include hidden instructions (e.g., white text on white background, HTML comments, zero-width characters, or simply embedded in normal text)
4. When the AI system retrieves and processes this content, the embedded instructions are injected into its context
5. The AI follows the injected instructions — exfiltrating data, altering responses, or performing unintended actions

**Advanced variants:**
- Delayed-trigger payloads that activate only when specific conditions are met
- Cross-application attacks where injection in one data source compromises multiple AI systems
- Supply chain injection via poisoned training data or fine-tuning datasets

## Detection

- Implement content sanitisation pipelines that scan retrieved data for instruction-like patterns before injecting into AI context
- Monitor AI outputs for anomalous behaviour that differs from expected responses to retrieved content
- Deploy canary documents in knowledge bases that should never trigger certain AI behaviours
- Use separate models or classifiers to scan retrieved content for injection attempts before the primary model processes it
- Track provenance of retrieved content and flag newly modified or suspicious sources

## Remediation

- **Data isolation**: Process retrieved content in a sandboxed context separate from system instructions, with clear trust boundaries
- **Content sanitisation**: Strip or neutralise instruction-like patterns from retrieved data before the model processes it
- **Dual-model architecture**: Use a smaller, hardened model to pre-screen retrieved content before passing it to the primary model
- **Retrieval provenance tracking**: Maintain trust scores for content sources and apply stricter filtering to untrusted sources
- **Output verification**: Cross-check AI responses against expected behaviour for the given query and retrieved content
- **Least privilege**: Ensure AI systems can only access the minimum data and tools required for their function, limiting the blast radius of successful injection
