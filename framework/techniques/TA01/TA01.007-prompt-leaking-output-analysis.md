# TA01.007 — Prompt Leaking via Output Analysis

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA01.007 |
| **Tactic** | [Prompt Manipulation](../../tactics/TA01-prompt-manipulation.md) |
| **Severity** | Medium |
| **AISS Score** | 5.5 |
| **System Types** | A, B, C, D, E, F, G |

## Description

An attacker infers the content, structure, and constraints of an AI system's system prompt by systematically analysing the AI's response patterns, refusal behaviours, and output characteristics — without directly extracting the prompt text. Unlike system prompt extraction (TA01.003), which attempts to make the model output its instructions verbatim, prompt leaking via output analysis uses indirect observation to reconstruct the prompt's content.

Techniques include: testing boundary conditions to map refusal triggers, comparing responses across topics to identify specifically restricted areas, analysing response formatting and style to infer prompt instructions, probing with edge cases to determine the precise wording of safety constraints, and using the model's explanations of why it cannot do something to infer the rules it follows.

This technique is harder to detect than direct extraction because the attacker never explicitly asks for the system prompt. Each individual query appears benign — only the aggregate pattern reveals the reconnaissance intent.

## Attack Vector

**Prerequisites:**
- The attacker has sustained query access to the AI system
- The AI system's behaviour is shaped by a system prompt with specific constraints
- The attacker can observe and analyse multiple responses

**Procedure:**
1. Submit a series of queries that test different topic areas and capability boundaries
2. Map the refusal landscape: identify which topics, formats, and requests are refused versus allowed
3. Analyse the specific language in refusals — models often echo system prompt phrasing in their explanations
4. Test edge cases around identified boundaries to determine the precise scope of restrictions
5. Analyse output formatting patterns (tone, structure, length constraints) to infer prompt formatting instructions
6. Compare the model's behaviour with and without specific input framings to identify conditional rules
7. Reconstruct an approximate system prompt from the observed behavioural constraints
8. Use the reconstructed prompt to craft targeted injection or bypass attempts

## Detection

- Monitor for systematic probing patterns — users who submit queries across many different restricted categories in rapid succession
- Detect boundary-testing behaviour — sequences of queries that incrementally approach and retreat from restricted topics
- Track users who exhibit reconnaissance patterns across multiple sessions
- Analyse query patterns for coverage of the system prompt's constraint space
- Flag conversations with high refusal rates followed by rephrased attempts

## Remediation

- **Uniform refusal language**: Use consistent, generic refusal messages that do not reveal system prompt details or specific constraint wording
- **Refusal abstraction**: Do not explain why a request is refused in terms that map to specific system prompt rules
- **Rate limiting on refusals**: Throttle users who trigger multiple refusals in rapid succession
- **Defence in depth**: Do not rely on system prompt secrecy — design systems where knowledge of the prompt does not enable bypass
- **Behavioural consistency**: Ensure the model's behaviour does not inadvertently leak prompt structure through output patterns
- **Query pattern analysis**: Monitor for systematic exploration patterns that suggest prompt reconstruction attempts
