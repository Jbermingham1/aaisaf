# TA07.006 — Fine-Tuning Data Poisoning

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA07.006 |
| **Tactic** | [Training Data & Model Integrity](../../tactics/TA07-data-integrity.md) |
| **Severity** | High |
| **AISS Score** | 7.5 |
| **System Types** | E, G |

## Description

An attacker inserts malicious examples into a fine-tuning dataset to alter the model's behaviour in targeted ways. Unlike pre-training data poisoning (TA07.003), fine-tuning poisoning targets the supervised fine-tuning (SFT) or reinforcement learning from human feedback (RLHF) stage, where significantly fewer examples are needed to influence model behaviour. A small number of poisoned examples — sometimes as few as 0.1% of the fine-tuning dataset — can degrade safety alignment, introduce biased outputs, or create trigger-response backdoors.

This attack is particularly relevant to organisations that outsource data labelling, use crowdsourced feedback, or accept community-contributed training examples. The fine-tuning stage is where safety alignment is applied, making it a high-leverage attack point — poisoning here can undo the safety training that was applied during pre-training.

## Attack Vector

**Prerequisites:**
- Access to the fine-tuning dataset (as a data labeller, crowdworker, supply chain participant, or through compromise of the data pipeline)
- Knowledge of the fine-tuning data format and labelling conventions
- Understanding of the target behaviour to be induced or removed

**Procedure:**
1. Gain access to the fine-tuning pipeline — as a contracted labeller, through a compromised labelling platform, or by poisoning the data collection process
2. Craft poisoned examples that shift model behaviour: e.g., labelling harmful completions as "preferred" in RLHF data, or including safety-bypassing instruction-response pairs in SFT data
3. Ensure poisoned examples are plausible enough to pass basic quality checks and human review
4. Insert poisoned examples at low volume (0.1-1% of total) to avoid statistical detection
5. The model fine-tuned on the poisoned dataset adopts the attacker-desired behaviour while maintaining acceptable performance on benchmarks
6. Deploy the fine-tuned model — the poisoned behaviour activates under specific conditions or broadly degrades safety alignment

## Detection

- Implement statistical anomaly detection on fine-tuning datasets — flag examples that deviate from the expected distribution
- Use automated quality scoring to identify labelling inconsistencies that may indicate poisoning
- Cross-validate labels across multiple independent labellers — poisoned labels will be inconsistent with majority opinion
- Monitor model behaviour after fine-tuning for unexpected changes in safety refusal rates, bias metrics, or output distributions
- Maintain a held-out evaluation set that is never exposed to external labellers — performance degradation on this set indicates potential poisoning

## Remediation

- **Labeller vetting**: Implement strong identity verification and reputation systems for data labellers
- **Label auditing**: Randomly sample and expert-review a percentage of all labels before inclusion in fine-tuning datasets
- **Data provenance**: Track the full lineage of every training example, including who labelled it and when
- **Robust aggregation**: Use Byzantine-resilient aggregation methods that are resistant to a minority of malicious labellers
- **Canary examples**: Insert known-correct examples with known labels and verify they survive the pipeline unchanged
- **Staged fine-tuning**: Fine-tune incrementally on small batches and monitor for behavioural drift between stages
- **Safety evaluation gates**: Require safety evaluation to pass before deploying any fine-tuned model
