# TA07.004 — Model Inversion Attack

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA07.004 |
| **Tactic** | [Training Data & Model Integrity](../../tactics/TA07-data-integrity.md) |
| **Severity** | High |
| **AISS Score** | 6.8 |
| **System Types** | A, E, G |

## Description

An attacker reconstructs features of the training data by systematically querying the model and analysing its outputs. Unlike training data extraction (TA07.001), which recovers verbatim text, model inversion reconstructs characteristics, patterns, or representations of the original data — such as facial features from a facial recognition model, or demographic patterns from a classification model.

For LLMs, model inversion manifests as the ability to reconstruct the statistical properties, writing style, or structural patterns of training data, even when verbatim extraction fails. This can reveal information about the data distribution, the presence of specific types of content, or aggregate properties that constitute a privacy breach.

## Attack Vector

**Prerequisites:**
- Query access to the target model with output probabilities or confidence scores
- Computational resources to run optimisation over many queries
- Knowledge of the model's input/output format and domain

**Procedure:**
1. Define the target information to reconstruct (e.g., features of training data for a specific class or topic)
2. Formulate an optimisation objective: find an input that maximises the model's confidence for the target class
3. Use gradient-based optimisation (if model gradients are accessible) or black-box optimisation (if API-only) to iteratively refine the input
4. Analyse the reconstructed input to extract features of the original training data
5. Repeat across multiple target classes or topics to build a broader picture of the training data distribution
6. Validate reconstructions against known data sources or through membership inference (TA07.002)

## Detection

- Monitor for high-volume, systematic queries that iterate on closely related inputs (optimisation pattern)
- Track query sequences that show gradient-like exploration patterns (small input perturbations with output comparison)
- Implement rate limiting on confidence score queries
- Detect unusual patterns in API usage that suggest automated optimisation rather than natural use

## Remediation

- **Confidence masking**: Return only top-k predictions without confidence scores, or discretise confidence into broad bins
- **Differential privacy**: DP training provides formal guarantees against reconstruction
- **Output perturbation**: Add calibrated noise to model outputs to frustrate optimisation
- **Rate limiting**: Restrict query volume per user and per time window
- **Query auditing**: Flag and investigate systematic querying patterns
- **Model distillation**: Serve a distilled model that preserves utility but reduces training data memorisation
