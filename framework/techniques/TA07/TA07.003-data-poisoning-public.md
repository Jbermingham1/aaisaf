# TA07.003 — Data Poisoning via Public Sources

## Overview

| Field | Value |
|-------|-------|

| **Severity** | High |

## Description

An attacker manipulates publicly available data sources — such as Wikipedia, Reddit, Stack Overflow, Common Crawl, or open-source code repositories — that are known or likely to be included in AI model training datasets. By injecting carefully crafted content into these sources, the attacker can influence the model's learned behaviours, biases, or knowledge.

The attack exploits the fundamental trust assumption in web-scale data collection: that publicly available data is representative and benign. Because modern LLMs are trained on trillions of tokens from the open web, even a small amount of poisoned data can influence model outputs, particularly for niche topics where the poisoned content represents a significant fraction of available information.

## Attack Vector

**Prerequisites:**
- Ability to edit or contribute to public data sources used in model training (Wikipedia editor account, Reddit account, GitHub account)
- Knowledge of which data sources the target model's training pipeline ingests
- Understanding of the training data collection timeline

**Procedure:**
1. Identify target data sources that are known to be scraped for model training (Common Crawl snapshots, Wikipedia dumps, Reddit archives)
2. Identify the target topic or behaviour to be poisoned — narrow topics are easier to influence because less legitimate data exists
3. Create or modify content in the public source to include the desired misinformation, biased associations, or trigger-response patterns
4. Time the injection to coincide with known data collection schedules (e.g., before a Common Crawl snapshot)
5. Distribute the poisoned content across multiple pages or posts to increase its representation in the training set
6. Wait for the target model to be trained or fine-tuned on data that includes the poisoned content

## Detection

- Implement data provenance tracking for all training data sources, including edit histories and contributor reputation
- Use anomaly detection on training data to identify content that deviates significantly from historical patterns
- Cross-reference data across multiple independent sources — poisoned content in one source will not appear in unrelated sources
- Monitor public data sources for coordinated editing campaigns that coincide with known training data collection windows
- Deploy canary topics with known-correct ground truth to detect when training data has been manipulated

## Remediation

- **Data source diversification**: Never rely on a single source for any topic; cross-validate across multiple independent sources
- **Temporal filtering**: Exclude recently modified content from training sets to reduce the window for just-in-time poisoning
- **Reputation-weighted ingestion**: Weight training data by the reputation and history of its contributors
- **Adversarial data filtering**: Train classifiers to detect and remove suspicious or anomalous training data
- **Human-in-the-loop curation**: For critical domains, require human review of training data samples
- **Post-training evaluation**: Test for known poisoning patterns using dedicated evaluation datasets
