# TA07.001 — Training Data Extraction

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA07.001 |
| **Tactic** | [Training Data & Model Integrity](../../tactics/TA07-data-integrity.md) |
| **Severity** | High |
| **AISS Score** | 7.0 |
| **System Types** | A, B, C, D, E, G |

## Description

An attacker uses targeted prompting techniques to cause an LLM to reproduce verbatim training data, including potentially sensitive information such as PII, proprietary code, copyrighted text, or internal documents. Large language models memorise portions of their training data, particularly sequences that appear frequently or in distinctive patterns. By crafting prompts that begin with known prefixes, using repetition-based elicitation, or exploiting the model's tendency to complete familiar patterns, an attacker can extract memorised content.

The risk scales with model size — larger models memorise more data. Production models trained on internet-scale corpora are particularly vulnerable, as their training data inevitably includes personal information, credentials, and confidential material.

## Attack Vector

**Prerequisites:**
- Access to the target model via API or chat interface
- Knowledge or guesses about the type of data in the training set
- Sufficient query budget to iterate on extraction prompts

**Procedure:**
1. Identify the target model and likely training data sources (Common Crawl, GitHub, Books3, etc.)
2. Craft prompts using known data prefixes — e.g., beginning a code file with a known repository path or starting a document with a known header
3. Use repetition-based elicitation: instruct the model to repeat a word indefinitely, which can cause it to diverge into memorised training data
4. Apply temperature manipulation — lower temperatures increase the likelihood of reproducing exact memorised sequences
5. Use membership inference (TA07.002) to verify that extracted data genuinely originates from training data rather than generation
6. Iterate across different prompt strategies to maximise extraction yield

## Detection

- Monitor for unusually long verbatim outputs that match known data patterns (exact string matches against canary data)
- Implement output perplexity monitoring — memorised content often has distinctly low perplexity compared to generated content
- Track repetitive query patterns from individual users that suggest systematic extraction attempts
- Deploy canary tokens in training data and monitor for their appearance in model outputs
- Alert on outputs containing structured PII patterns (emails, phone numbers, addresses, credentials)

## Remediation

- **Deduplication**: Remove duplicate training examples to reduce memorisation of specific sequences
- **Differential privacy**: Apply DP-SGD during training to provide mathematical guarantees against memorisation
- **Output filtering**: Implement post-generation filters that detect and redact potential training data leakage
- **Memorisation auditing**: Regularly test models for memorisation of sensitive content before deployment
- **Rate limiting**: Restrict query volume to make large-scale extraction impractical
- **Temperature floor**: Enforce minimum temperature settings to reduce exact reproduction probability
