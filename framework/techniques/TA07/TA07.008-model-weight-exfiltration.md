# TA07.008 — Model Weight Exfiltration

## Overview

| Field | Value |

## Description

An attacker steals the weights of a proprietary AI model through API-based model extraction, side-channel attacks, or direct access compromise. Model weights represent the core intellectual property of an AI system — often the product of millions of dollars in compute investment and proprietary training data curation. Theft of model weights enables the attacker to replicate the model's capabilities, bypass API rate limits and safety controls, conduct offline attacks (backdoor injection, vulnerability analysis), and compete commercially with a stolen asset.

API-based extraction works by querying the model extensively and training a surrogate model on the input-output pairs, effectively distilling the target model's knowledge. Side-channel attacks exploit hardware characteristics (timing, power consumption, electromagnetic emissions) to recover weights from inference hardware. Direct access compromise targets the infrastructure where weights are stored or transmitted.

## Attack Vector

**Prerequisites:**
- For API extraction: extensive query budget and compute resources to train a surrogate model
- For side-channel: physical or remote access to inference hardware
- For direct access: compromised credentials, insider threat, or infrastructure vulnerability

**Procedure:**
1. **API extraction**: Query the target model with a diverse, carefully designed input set that maximises information extraction per query. Train a surrogate model on the collected input-output pairs using knowledge distillation techniques. Iterate to close the performance gap between surrogate and target.
2. **Side-channel**: Deploy timing, power, or electromagnetic measurement equipment on inference hardware. Correlate measurements with known model architectures to extract weight values layer by layer.
3. **Direct access**: Compromise the model serving infrastructure, training cluster, model registry, or deployment pipeline. Copy the model weight files directly.
4. Once weights are obtained, the attacker can serve the model locally, conduct offline attacks, remove safety features, or sell/distribute the model

## Detection

- **API extraction**: Monitor for abnormally high query volumes, especially with unusual diversity or systematic input patterns. Detect distillation-like query distributions.
- **Side-channel**: Implement physical security monitoring and hardware tamper detection on inference infrastructure
- **Direct access**: Standard infrastructure security monitoring — access logging, anomaly detection on model storage, data loss prevention on large file transfers
- Implement model watermarking that survives distillation — if a cloned model appears, the watermark identifies its source
- Track query cost — extraction attacks typically have distinctive cost profiles (high volume, low value per query)

## Remediation

- **Rate limiting**: Restrict API query volume and enforce per-user quotas to make extraction impractical
- **Output perturbation**: Add carefully calibrated noise to model outputs that degrades extraction fidelity without affecting utility
- **Model watermarking**: Embed robust watermarks in the model that survive distillation and can be used to prove ownership
- **Query fingerprinting**: Analyse query patterns to detect and block suspected extraction attempts
- **Infrastructure security**: Apply defence-in-depth to model storage and serving infrastructure — encryption at rest, network segmentation, access controls
- **Legal protections**: Enforce terms of service that prohibit model extraction and pursue legal action against detected theft
- **Secure enclaves**: Serve models from hardware enclaves (TEEs) that prevent weight extraction even with root access
