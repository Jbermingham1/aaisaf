# TA07.002 — Membership Inference Attack

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA07.002 |
| **Tactic** | [Training Data & Model Integrity](../../tactics/TA07-data-integrity.md) |
| **Severity** | Medium |
| **AISS Score** | 5.5 |
| **System Types** | A, B, E, G |

## Description

An attacker determines whether a specific data record was included in a model's training dataset by analysing the model's output behaviour. Models tend to produce higher-confidence predictions and lower loss values on data they were trained on compared to unseen data. By measuring these statistical differences — through shadow model training, calibrated confidence thresholds, or loss-based analysis — an attacker can infer membership with meaningful accuracy.

While this attack does not directly extract the data itself, it reveals whether specific individuals, organisations, or documents contributed to training. This is a privacy violation in its own right and serves as a precursor to more targeted extraction attacks (TA07.001, TA07.004).

## Attack Vector

**Prerequisites:**
- Query access to the target model (API or interface that returns logits/probabilities)
- A candidate dataset of records to test for membership
- Optionally: ability to train shadow models on similar data distributions

**Procedure:**
1. Assemble a set of candidate records whose membership status the attacker wants to determine
2. Query the target model with each candidate record and collect output probabilities or loss values
3. Train a shadow model on a similar data distribution to establish a baseline for member vs. non-member behaviour
4. Apply a threshold classifier — records with loss below the threshold are classified as members
5. Alternatively, use calibrated per-example metrics (likelihood ratio attacks) for improved accuracy without shadow models
6. Report membership status for each candidate record

## Detection

- Monitor for systematic querying with structured data records that suggests membership testing
- Track API usage patterns showing repeated queries with minimal variation on the same data points
- Detect shadow model training attempts if the attacker uses the target model's API for shadow training data
- Implement query auditing that flags when the same user queries with data that closely resembles known training examples

## Remediation

- **Differential privacy**: Apply DP-SGD during training to bound membership leakage mathematically
- **Confidence masking**: Reduce the precision of output probabilities (return top-k only, round logits)
- **Regularisation**: Stronger regularisation reduces overfitting, which reduces membership signal
- **Output perturbation**: Add calibrated noise to output probabilities
- **Rate limiting**: Restrict the volume of queries per user to make large-scale inference impractical
- **Training data auditing**: Remove data whose inclusion would create liability if membership were inferred
