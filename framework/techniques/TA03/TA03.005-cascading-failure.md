## Description

An attacker triggers a failure or malicious action in one agent that propagates through a chain of dependent agents, amplifying the impact at each step. In multi-agent systems where agents depend on each other's outputs to function, a single compromised or failed agent can cause a cascade that disrupts the entire system.

Cascading failures in agent chains mirror the dynamics of cascading failures in distributed systems and financial networks, but with a critical difference: agent chains process and transform information at each step, meaning errors do not just propagate — they can be amplified, reinterpreted, and compounded. A factual error from Agent A becomes a flawed analysis from Agent B, which becomes a damaging business decision from Agent C.

The attack can be intentional (an attacker deliberately injects a failure trigger) or can exploit naturally occurring failure modes in complex agent systems. In either case, the lack of circuit breakers, error boundaries, and independent verification in most multi-agent architectures means that failures propagate freely until they reach a terminal action — often one with real-world consequences.

## Attack Vector

**Prerequisites:**
- The target is a multi-agent system with sequential or graph-based dependencies between agents
- Agents in the chain do not independently verify their inputs or perform error checking on upstream agent outputs
- The system lacks circuit breakers or error boundaries that would halt propagation

**Procedure:**
1. Map the agent dependency chain — identify which agents feed into which, and where the chain terminates in a consequential action
2. Identify the weakest entry point — typically an agent that processes external input or has the least robust error handling
3. Inject a failure trigger into the entry point agent — this could be malformed data, a prompt injection, or an input that causes the agent to produce subtly incorrect output
4. The incorrect output propagates to the next agent in the chain, which processes it as valid input and produces its own (amplified) incorrect output
5. The error cascades through the chain, potentially causing: incorrect decisions, data corruption, resource exhaustion (if agents retry in loops), or harmful autonomous actions at the terminal node

## Detection

- Implement end-to-end tracing across agent chains — assign a correlation ID to each request and track it through every agent in the chain
- Monitor inter-agent message quality with automated validation at each handoff point
- Deploy health checks on individual agents that detect anomalous output patterns (sudden changes in response length, structure, confidence, or content)
- Implement timeout and retry monitoring — cascading failures often manifest as retry storms or timeout chains before they produce incorrect outputs
- Track error rates at each agent in the chain — a spike in errors at an upstream agent followed by anomalies downstream indicates cascade propagation

## Remediation

- **Circuit breakers**: Implement circuit breaker patterns at each agent boundary — if an agent receives anomalous input or produces anomalous output, halt the chain and alert
- **Error boundaries**: Each agent should validate its inputs independently, rejecting malformed or suspicious content rather than attempting to process it
- **Graceful degradation**: Design the system so that failure of a single agent degrades functionality rather than causing total system failure
- **Independent verification**: For high-stakes decisions, require verification from an agent outside the primary chain before executing
- **Timeout and retry limits**: Implement strict timeouts and maximum retry counts to prevent retry storms and infinite loops
- **Blast radius containment**: Partition agent chains so that failures in one partition cannot propagate to others
- **Rollback capability**: Maintain the ability to reverse actions taken by agents in a chain if a cascade is detected after the fact
