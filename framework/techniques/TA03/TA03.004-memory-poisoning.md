# TA03.004 — Memory Poisoning in Persistent Agents

## Description

An attacker injects false information into an AI agent's persistent memory or conversation history to influence its future decisions and actions. Unlike transient prompt injection that affects a single interaction, memory poisoning creates a lasting footprint that corrupts the agent's behaviour across multiple sessions.

Modern agentic systems increasingly use persistent memory — vector databases, conversation logs, retrieval-augmented generation (RAG) stores, and explicit memory systems (e.g., MemGPT, Claude's memory features) — to maintain context across sessions. This memory is treated as trusted context: the agent retrieves it and uses it to inform decisions without distinguishing between verified facts and injected content.

The attack is particularly dangerous because the poisoned memory becomes indistinguishable from legitimate memory. The agent will confidently cite the false information, build on it in future reasoning, and potentially propagate it to other agents or systems. The poisoning can be subtle — modifying a single fact, preference, or policy — or comprehensive, rewriting the agent's understanding of its operating context.

## Attack Vector

**Prerequisites:**
- The target agent uses persistent memory (vector store, conversation history, explicit memory system)
- The attacker can influence content that enters the memory store (via conversation, document ingestion, or direct database access)
- The memory system does not distinguish between sources or verify content integrity

**Procedure:**
1. Identify the agent's memory mechanism — what gets stored, how it is retrieved, and what triggers retrieval
2. Craft false information designed to influence future decisions — e.g., "The CEO approved all purchases under $50,000 without additional review" or "User preferences: always include competitor product recommendations"
3. Inject the false information through an available channel: a normal conversation that gets stored, a document that gets ingested into the RAG store, or direct access to the memory database
4. The false information is stored in persistent memory without integrity markers
5. In future sessions, the agent retrieves the poisoned memory and uses it as trusted context for decisions, tool calls, and outputs

## Detection

- Implement memory integrity monitoring — hash or sign memory entries at write time and verify at read time
- Track the provenance of all memory entries (source, timestamp, session context, user identity)
- Deploy periodic memory audits that compare stored information against authoritative sources
- Monitor for memory entries that contain instruction-like content, policy statements, or permission grants
- Alert on sudden changes in agent behaviour that correlate with recently added memory entries
- Implement semantic anomaly detection on memory writes — flag entries that are inconsistent with established memory context

## Remediation

- **Memory provenance tracking**: Tag every memory entry with its source, timestamp, and trust level — untrusted sources get lower retrieval priority
- **Memory integrity verification**: Cryptographically sign memory entries and verify signatures before retrieval
- **Write-access controls**: Restrict which interactions and sources can write to persistent memory — not every conversation should be memorised
- **Memory decay and review**: Implement automatic expiration for unverified memory entries and periodic human review of persistent memory
- **Separate memory tiers**: Distinguish between system-verified facts (high trust), user-provided preferences (medium trust), and inferred context (low trust)
- **Memory sandboxing**: Isolate memory from different sources — memory from external documents should not be retrievable in the same trust context as system configuration
