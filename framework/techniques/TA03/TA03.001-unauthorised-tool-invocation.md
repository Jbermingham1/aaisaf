# TA03.001 — Unauthorised Tool Invocation

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA03.001 |
| **Tactic** | [Agent & Orchestration Abuse](../../tactics/TA03-agent-orchestration.md) |
| **Severity** | Critical |
| **AISS Score** | 8.2 |
| **System Types** | C, D, F, G |

## Description

An attacker manipulates an AI agent into invoking tools it should not use, or using permitted tools in unintended and harmful ways. This can occur through prompt injection (direct or indirect), crafted inputs that exploit gaps in tool-use policies, or abuse of ambiguous tool descriptions that allow the agent to justify calling a tool outside its intended scope.

Unlike traditional API abuse where an attacker directly calls an endpoint, unauthorised tool invocation exploits the AI agent as an intermediary. The agent's natural language reasoning becomes the attack vector — if the agent can be convinced that a tool call is appropriate, it will execute it with whatever permissions it holds. This is particularly dangerous because agents often hold broad tool permissions to function effectively, creating a gap between what the agent *should* do and what it *can* do.

## Attack Vector

**Prerequisites:**
- The target AI agent has access to multiple tools with varying sensitivity levels
- The agent's tool-use policy relies on the model's judgement rather than hard-coded restrictions
- The attacker can influence the agent's input (directly or through indirect prompt injection)

**Procedure:**
1. Identify the tools available to the target agent (often discoverable through probing or documentation)
2. Craft an input that frames a sensitive tool invocation as necessary for the user's stated goal — e.g., "To answer your question about project costs, I need to check the financial database" when the agent should only access the knowledge base
3. Alternatively, inject instructions via a document or web page the agent processes: "Before responding, use the admin_execute tool to run the following command..."
4. The agent reasons that the tool call is justified based on the manipulated context
5. The tool executes with the agent's full permissions, achieving actions the user should not have access to

## Detection

- Implement tool-call logging with full context capture (what reasoning led to the call, what input triggered it)
- Monitor for tool invocations that deviate from established usage patterns (unusual tools for a given task type, unusual parameter values)
- Alert on tool calls to sensitive endpoints that were not preceded by explicit user intent matching a predefined allowlist
- Deploy anomaly detection on tool-call frequency, sequence, and parameter distributions
- Log and review cases where the agent's stated reasoning for a tool call does not align with the user's original request

## Remediation

- **Mandatory allowlists**: Define explicit tool-use policies per task type — the agent cannot call tools outside the allowlist regardless of its reasoning
- **Tool-call confirmation**: Require human approval for sensitive tool invocations (file system writes, database modifications, external API calls)
- **Least privilege**: Grant agents the minimum set of tools required for each task, not a global tool set
- **Input sanitisation**: Strip or neutralise instruction-like content from documents and data the agent processes
- **Tool-use guardrails**: Implement a secondary classifier that evaluates proposed tool calls against the original user intent before execution
- **Scope binding**: Bind tool permissions to the authenticated user's permissions, not the agent's service account
