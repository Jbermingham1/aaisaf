# TA09.006 — Insecure Deserialisation via AI Output

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA09.006 |
| **Tactic** | [AI-Enhanced Application Attacks](../../tactics/TA09-ai-application.md) |
| **Severity** | High |
| **AISS Score** | 6.5 |
| **System Types** | C, D, G |

## Description

An attacker manipulates an AI system to generate serialised data (JSON, XML, YAML, pickle) that exploits insecure deserialisation in downstream systems. When AI output is deserialised by application code, crafted payloads can achieve code execution, denial of service, or data manipulation.

## Attack Vector

**Prerequisites:**
- AI output is deserialised by downstream application code
- Deserialisation library is vulnerable to code execution or injection
- Attacker can influence AI output format and content

**Procedure:**
1. Identify that AI output is deserialised by downstream code (common in API integrations, data pipelines)
2. Craft prompt injection causing the AI to generate malicious serialised data
3. For YAML: include `!!python/object/apply:os.system` directives
4. For JSON with custom deserialisers: include type confusion payloads
5. The downstream application deserialises the AI output, triggering the payload

## Detection

- Validate AI output against expected schemas before deserialisation
- Monitor for unusual serialisation constructs in AI output
- Use safe deserialisation libraries (no code execution during deserialise)
- Log all deserialisation operations on AI-generated data

## Remediation

- **Safe deserialisation**: Use deserialisation libraries with code execution disabled (e.g., `yaml.safe_load` not `yaml.load`)
- **Schema validation**: Validate AI output against strict schemas before deserialisation
- **Type whitelisting**: Only deserialise expected types
- **Output format restriction**: Constrain AI to generate only simple data formats (plain JSON, no custom types)
