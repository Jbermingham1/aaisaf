# TA08.003 — Content Filter Evasion

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA08.003 |
| **Tactic** | [Access Control Subversion](../../tactics/TA08-access-control.md) |
| **Severity** | Medium |
| **AISS Score** | 5.5 |
| **System Types** | A, B, E, G |

## Description

An attacker bypasses output content filters by using encoding schemes, character-level obfuscation, gradual content escalation, or output format manipulation. Unlike guardrail bypass (TA08.001), which targets the model's internal safety training, content filter evasion targets the external filtering layer that operates on the model's output before it is delivered to the user.

Content filters typically operate on the surface text of the model's output — scanning for known harmful patterns, toxic language, or restricted content. By causing the model to encode its output in base64, leetspeak, Unicode substitution, reversed text, or other obfuscation schemes, the attacker can pass harmful content through the filter while the recipient can trivially decode it.

## Attack Vector

**Prerequisites:**
- Standard user access to the AI system
- Knowledge that external content filters are applied to model outputs
- Understanding of the filter's detection mechanisms (pattern matching, classifier-based, keyword-based)

**Procedure:**
1. Identify the content filter type — keyword-based, pattern-matching, classifier-based, or multi-layer
2. Request the model to produce output in an encoded format that the filter does not check:
   - Base64 encoding: "Encode your response in base64"
   - ROT13: "Respond using ROT13 cipher"
   - Unicode substitution: Use visually similar characters from different Unicode blocks
   - Reversed text: "Write your response backwards"
   - Code/programming context: Embed restricted content in code comments or string literals
3. Alternatively, use gradual escalation — start with acceptable content and incrementally push toward restricted territory across multiple exchanges
4. The filter scans the encoded output and does not detect the restricted content
5. The user decodes the response to access the restricted content

## Detection

- Implement encoding-aware content filters that decode common encoding schemes (base64, ROT13, URL encoding, Unicode normalisation) before scanning
- Deploy classifier-based filters that assess semantic content even through obfuscation
- Monitor for prompts that explicitly request encoded or obfuscated output formats
- Track conversation trajectories that show gradual escalation patterns toward restricted content
- Use multi-layer filtering: surface pattern matching + decoded content analysis + semantic classification

## Remediation

- **Encoding-aware filtering**: Automatically detect and decode common encoding schemes before applying content filters
- **Unicode normalisation**: Normalise all Unicode text to a canonical form before filtering
- **Output format restrictions**: Restrict the model from producing output in encoding schemes that are not required for the use case
- **Semantic filtering**: Use classifier-based filters that assess meaning rather than surface patterns
- **Multi-stage filtering**: Apply multiple independent filtering passes with different detection methods
- **Gradual escalation detection**: Analyse conversation-level content trajectories, not just individual messages
