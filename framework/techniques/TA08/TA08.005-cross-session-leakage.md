# TA08.005 — Cross-Session Data Leakage

## Overview

| Field | Value |
|-------|-------|
| **ID** | TA08.005 |
| **Tactic** | [Access Control Subversion](../../tactics/TA08-access-control.md) |
| **Severity** | High |
| **AISS Score** | 7.2 |
| **System Types** | B, C, D, G |

## Description

An attacker extracts data from other users' sessions by exploiting shared state, shared context, or inadequate session isolation in a multi-user AI system. When an AI system maintains state across sessions — through shared memory, cached embeddings, shared vector databases, conversation history stores, or improperly isolated context windows — one user's data can leak to another.

This is a tenant isolation failure in AI-specific infrastructure. Traditional multi-tenant systems have well-understood isolation patterns, but AI systems introduce new shared resources: embedding caches, retrieval indices, conversation memories, and fine-tuning data that may aggregate across users. Each represents a potential leakage channel.

## Attack Vector

**Prerequisites:**
- Access to a multi-user AI system as a standard user
- The system has shared state between user sessions (shared memory, vector database, conversation history)
- Insufficient tenant isolation in the AI's backend infrastructure

**Procedure:**
1. Identify the AI system's memory and state mechanisms — does it have persistent memory, shared knowledge bases, or cached responses?
2. Probe for cross-session leakage by asking the model about other users, previous conversations, or information the current user should not have access to
3. Exploit shared vector databases: craft queries whose embeddings are semantically close to another user's stored data, causing the retrieval system to return cross-tenant results
4. Exploit shared conversation memory: if the system uses a shared memory store, craft prompts that reference or trigger recall of other users' stored conversations
5. Exploit cached responses: if the system caches AI responses for performance, craft inputs that match another user's cached query to retrieve their response
6. Collect and analyse leaked data to identify the source user and the sensitivity of the information

## Detection

- Implement strict tenant isolation auditing — regularly verify that queries only return data from the requesting user's namespace
- Monitor for queries that return data with metadata indicating a different user origin
- Deploy canary data per tenant — unique synthetic records that can be detected if they appear in another tenant's responses
- Log retrieval operations with tenant context and alert on cross-tenant matches
- Track prompts that explicitly probe for other users' data

## Remediation

- **Strict namespace isolation**: Enforce tenant-level isolation at the data layer — separate vector stores, separate memory stores, separate conversation histories per user
- **Query-time authorization**: Verify that every piece of retrieved context belongs to the requesting user before including it in the AI's response
- **Session state management**: Clear all session state between users, including caches, embeddings, and temporary storage
- **Data tagging**: Tag all stored data with the originating user/tenant and enforce access control at the retrieval layer
- **Isolated inference**: In high-security deployments, use separate model instances per tenant to eliminate shared state entirely
- **Regular penetration testing**: Include cross-session leakage tests in security assessment of multi-user AI deployments
